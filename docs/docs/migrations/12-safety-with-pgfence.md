# Migration safety checks with pgfence

TypeORM migrations are plain SQL inside `queryRunner.query()` calls. They run
against your production database, often without anyone reading the generated
SQL line by line. A single `ALTER COLUMN ... TYPE` or `CREATE INDEX` without
`CONCURRENTLY` can block every read and write on a hot table for the duration
of the rewrite.

[pgfence](https://pgfence.com) is an open-source CLI that parses TypeORM
migration files, extracts the SQL from each `queryRunner.query()` call, and
reports the Postgres lock mode each statement acquires, the risk level, and
the safe rewrite for any pattern that locks the table.

This page describes how to run pgfence against TypeORM migrations locally and
in CI.

## Install

```shell
npm install --save-dev @flvmnt/pgfence
```

Or run it ad-hoc without installing:

```shell
npx @flvmnt/pgfence analyze --format typeorm src/migrations/*.ts
```

## Local check

After running `typeorm migration:generate`, point pgfence at the generated
file:

```shell
npx @flvmnt/pgfence analyze --format typeorm src/migrations/1700000000000-AddEmailToUser.ts
```

For a generated migration that adds a `NOT NULL` column on an existing table,
pgfence reports something like:

```
src/migrations/1700000000000-AddEmailToUser.ts:5  HIGH    ACCESS EXCLUSIVE
  ALTER TABLE "user" ADD "email" character varying NOT NULL

  Adding a NOT NULL column without a DEFAULT rewrites the entire table under
  ACCESS EXCLUSIVE. Reads and writes block until it finishes.

  Safe rewrite:
    1. ADD COLUMN nullable
    2. backfill in batches
    3. ADD CONSTRAINT ... CHECK NOT VALID + VALIDATE CONSTRAINT
    4. ALTER COLUMN SET NOT NULL

Analyzed 1 SQL statement. 0 dynamic statements not analyzable.
```

The output names the lock mode (ACCESS EXCLUSIVE), what it blocks, and the
expand/backfill/contract sequence that achieves the same end state without
locking the table.

## Risk levels and exit codes

| Risk | Meaning | Examples |
|---|---|---|
| SAFE | No table-locking effects | `ADD COLUMN` nullable, `ALTER COLUMN SET DEFAULT` |
| LOW | Brief metadata-only lock | `RENAME COLUMN` on PG14+ |
| MEDIUM | Blocks writes during the operation | `CREATE INDEX` without `CONCURRENTLY` |
| HIGH | Blocks reads and writes | `ALTER COLUMN TYPE`, `ADD COLUMN NOT NULL` without `DEFAULT` |
| CRITICAL | Irreversible or destructive | `DROP TABLE`, `TRUNCATE`, `DROP DATABASE` |

In `--ci` mode, pgfence exits non-zero when a finding meets or exceeds the
threshold set by `--max-risk`. Combine with a CI workflow to fail the build
on dangerous patterns before they reach production.

## CI: GitHub Actions

```yaml
# .github/workflows/migration-safety.yml
name: Migration safety
on:
  pull_request:
    paths:
      - "src/migrations/**"

jobs:
  pgfence:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - name: Analyze migrations
        run: npx --yes @flvmnt/pgfence analyze --format typeorm --ci --max-risk medium src/migrations/*.ts
```

The workflow fails the PR on any MEDIUM, HIGH, or CRITICAL finding. Lower the
threshold to `high` while you adopt the tool, then raise it once your
existing migrations are clean.

## Auto-detect

pgfence detects TypeORM by content, so `--format typeorm` is optional:

```shell
npx @flvmnt/pgfence analyze src/migrations/*.ts
```

The detector recognises `queryRunner.query()` calls and treats each string
literal as a separate analyzable statement. Statements built from template
literals or computed identifiers are reported as `UNKNOWN` (counted in the
coverage line) rather than skipped silently.

## Database-size aware risk scoring

pgfence can adjust risk levels based on table sizes. Provide a stats file
exported from a read replica:

```shell
npx @flvmnt/pgfence analyze \
  --format typeorm \
  --stats-file pgfence-stats.json \
  src/migrations/*.ts
```

A 1,000 row table and a 100M row table have very different blast radii for
the same `ALTER`. With stats loaded, pgfence escalates the risk for large
tables and downgrades it for small ones.

The stats file is generated by `npx @flvmnt/pgfence extract-stats` against
your read replica. pgfence itself never connects to your database.

## When in doubt

Ask pgfence to explain any statement:

```shell
npx @flvmnt/pgfence explain "ALTER TABLE \"user\" ALTER COLUMN \"email\" TYPE text"
```

It returns the lock mode, what it blocks, and the safe rewrite recipe.
